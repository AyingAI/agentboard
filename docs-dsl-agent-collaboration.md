# DSL 与 Agent 协作：为什么它值得关注

你可以先把 DSL 理解成一句话：

**DSL 是一种为特定任务设计的“小语言”，它把人的想法变成机器能稳定理解、校验、修改、渲染的结构。**

这里的 DSL 是 Domain Specific Language，中文可以叫“领域专用语言”。它不是一种通用编程语言，不是 Python、JavaScript 那种什么都能干的语言；它通常只服务一个明确场景，比如画图、排版、工作流、表单、自动化、数据看板、UI 布局。

## 1. 为什么不用自然语言直接让 Agent 画图

自然语言很适合表达意图，但它有三个天然问题：

1. **容易含糊**  
   你说“画一个清楚的流程图”，Agent 需要猜：几层？几个节点？箭头怎么连？颜色是否有含义？

2. **不容易校验**  
   一段自然语言很难自动检查“节点是否重叠”“文字是否溢出”“连线是否引用了不存在的节点”。

3. **不容易复用和局部修改**  
   如果你说“把第三步改成紫色，右侧说明区加一个风险”，Agent 必须重新理解整张图，容易牵一发动全身。

DSL 的价值，就是在自然语言和最终画板之间加一层“结构化中间表示”。

```
人的自然语言想法
   -> Agent 理解和规划
   -> DSL：结构化图元、布局、颜色、连线
   -> 渲染器 / whiteboard-cli / 前端组件
   -> 人能看到、能编辑的可视化结果
```

## 2. DSL 产出“结构化图元”到底是什么意思

以飞书画板为例，Agent 不只是生成一张图片，而是生成一组结构化对象：

- frame：一个分组、容器、区域
- rect：一个节点、卡片、模块
- text：一段文字
- connector：一条连线
- style：颜色、边框、字体
- layout：横排、竖排、绝对定位、拓扑布局

所以最终画板不是“死图”，而是一堆可编辑的对象。你可以在飞书画板里继续拖动、改字、改颜色。

这点很重要：**DSL 让 Agent 的输出从“结果图片”变成“可继续协作的工作对象”。**

## 3. DSL 是否有助于 Agent 理解

答案是：**有帮助，但前提是 DSL 设计得足够清晰。**

对 Agent 来说，DSL 有点像一种“外部工作记忆”。它把 Agent 脑子里的理解固定下来，变成可读、可检查、可继续推理的结构。

比如自然语言里的一句话：

“用户先提出需求，Agent 生成 DSL，再渲染成画板，人类继续修改。”

在 DSL 或结构化 JSON 里可以拆成：

```json
{
  "nodes": [
    { "id": "human", "label": "人提出需求" },
    { "id": "agent", "label": "Agent 理解和规划" },
    { "id": "dsl", "label": "生成 DSL" },
    { "id": "canvas", "label": "渲染成白板" }
  ],
  "edges": [
    ["human", "agent"],
    ["agent", "dsl"],
    ["dsl", "canvas"]
  ]
}
```

这样 Agent 后续就不需要重新猜图里有什么，而是能直接读取结构：有哪些节点、节点之间是什么关系、哪个节点需要修改。

## 4. 它为什么能提高人与 Agent 的协作效率

我认为关键不只是“Agent 能画图”，而是协作方式会变：

### 过去的协作方式

人说一句话，Agent 生成一个结果。  
结果不满意，人继续用自然语言描述修改。  
Agent 重新猜一遍，常常改 A 影响 B。

### DSL 参与后的协作方式

人表达目标，Agent 生成 DSL 和可视化结果。  
人看到图，直接指出哪里不对，或者在白板上改。  
Agent 读取结构化图元，知道具体改哪个节点、哪条关系、哪个分组。

这就从“反复聊天纠偏”变成了“围绕同一个结构化对象共同编辑”。

## 5. 人画白板，Agent 再理解，是否可行

可行，而且这是一个很有价值的方向。

理想闭环是：

1. 人用白板画出脑中的想法  
   不需要懂 DSL，只需要按自己的方式画框、箭头、分组、便签。

2. 系统把白板解析成结构  
   识别节点、文字、箭头、分组、空间关系。

3. 转成 DSL 或结构化图模型  
   例如 nodes、edges、groups、layout、metadata。

4. Agent 基于结构理解  
   它不只是“看图”，而是知道这是一个流程、架构、因果链或决策树。

5. Agent 给出建议或重构  
   比如补漏、拆层、标风险、生成 PRD、生成任务、生成流程文档。

这背后的核心价值是：**白板成为人和 Agent 共享的外部大脑。**

## 6. 不止飞书画板，DSL 可以用于很多场景

| 场景 | DSL 可以表达什么 | 价值 |
|---|---|---|
| 白板 / 流程图 | 节点、连线、分组、布局 | 把想法变成可编辑图 |
| UI 设计 | 页面、组件、状态、交互 | Agent 可生成前端或设计稿 |
| 工作流自动化 | 触发器、条件、动作、异常处理 | 可视化搭建自动化 |
| 数据看板 | 指标、维度、图表、筛选器 | 把分析结果变成可复用仪表盘 |
| 表单系统 | 字段、校验、显隐规则 | 非技术人员也能配置业务系统 |
| Agent 编排 | 工具、步骤、上下文、失败回退 | 让多步 Agent 行为可观察 |
| 规则引擎 | 条件、动作、优先级 | 把业务规则从代码里抽出来 |
| 产品文档 | 模块、流程、依赖、风险 | 文档和系统结构可互相转换 |

所以，DSL 不是“画图专用技巧”，而是一种更通用的人机协作方式。

## 7. 对产品经理来说，最值得关注的点

如果从产品角度看，DSL 代表一种很关键的产品形态变化：

**AI 不再只是聊天框里的回答者，而是结构化工作对象的生成者、解释者和维护者。**

这意味着很多产品可以从“用户和 AI 对话”升级为：

- 用户说出目标
- AI 生成结构
- 前端渲染结构
- 用户在界面里直接编辑
- AI 继续理解编辑后的结构
- 系统把结构落到工作流、文档、代码、任务或数据里

这个闭环比纯聊天更强，因为它让协作有了共同对象。

## 8. 但 DSL 也不是万能的

DSL 的风险也很明确：

1. **DSL 设计太复杂，用户和 Agent 都会累**  
   如果 schema 太大、规则太多，Agent 更容易写错，人也更难理解。

2. **DSL 只适合结构明确的场景**  
   流程、架构、表单、规则、看板很适合；高度艺术化、情绪化、开放式创意不一定适合。

3. **渲染器很关键**  
   DSL 本身只是结构，最终体验取决于渲染器是否稳定、好看、可交互。

4. **需要可逆链路**  
   只支持“DSL -> 可视化”还不够，最好还支持“可视化编辑 -> DSL”，这样人和 Agent 才能真正双向协作。

## 9. 一个产品判断

我会把 DSL 看成 Agent 产品里的“中间层协议”。

它的价值不是让普通用户学习 DSL，而是让系统内部有一个稳定结构：

- 人用自然语言或白板表达
- Agent 转成 DSL
- 前端把 DSL 渲染成可视化对象
- 人继续编辑对象
- Agent 再读回结构并继续工作

最终，用户感知到的不是 DSL，而是：**我随手画出来、说出来的想法，AI 真的理解了，而且还能持续和我一起改。**

这就是 DSL 在 Agent 协作里的真正价值。
